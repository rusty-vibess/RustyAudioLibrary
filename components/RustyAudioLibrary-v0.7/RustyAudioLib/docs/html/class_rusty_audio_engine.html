Detailed Description

Handles all run-time execution of audio code.

From managing the signal chain to initialising the IO with device drivers. This class handles all things audio.

Makes use of PortAudio to open IO streams to devices and to create an audioCallback() on a high priority interrupt thread.

RustyAudioEngine cannot be instantiated more than once due to the audioCallback from PortAudio's IO stream being static.

Instantiated inside RustyAudioLibAPI as a private member.

Static audio callback.

Static audio callback used as workaround. In current implementation (and seemingly all IO device libraries) the callback function is written in C meaning it is static. This meant I was unable to call members from my non-static RustyAudioEngine class. This wrapper layer allowed for a workaround, by passing a reference to the static function I could essentially trick it into calling non-static member functions. I then use that to call the other audioCallback() function and process audio there.

Adds a RustyProcessor to signal chain.

Allows user to pass in a pointer to any object that inherits from RustyProcessor. Processor will then be added to the signal chain via the _processingUnit vector.

Realised later on that the template wasn't necessary and I could just use RustyProcessor as the argument but I like static_assert macro to check the type so kept this function as is.
